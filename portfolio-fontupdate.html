Component Architecture

The button: a foundation that reveals everything

A button is a deceptively simple component, but it is a foundational building block of any design system. It carries brand identity, drives user interaction, and appears in nearly every screen in the product.

This case study examines the button through the lens of form design and accessibility. Drawing on GOV.UK's high-contrast focus ring approach, Adam Silver's Form Design Patterns, and Nathan Curtis's state documentation principles, it traces how I approach component structure and behavior — with accessibility as the throughline.

Before touching Figma or code, I map the semantic layer. A button doesn't reference a color — it references a decision. That separation is what makes themes, dark mode, and brand refreshes non-destructive.

Why this matters: When the brand refreshes from teal to blue, we update one primitive token. Every component that consumes color.action.primary inherits the change without a single design file edit.

Every state is an accessibility decision. I design states not just for visual differentiation but for keyboard users, screen reader users, and users with low vision — people who depend on a visible, high-contrast focus indicator to know where they are in a form.

Key insight: Shared semantic tokens sit in a platform-agnostic JSON source of truth (Style Dictionary). Platform-specific transformers output CSS variables, Swift enums, and XML attributes from the same token file — ensuring a brand change propagates everywhere without manual translation.

Buttons don't live in isolation — they live at the end of forms. The submit button is the most consequential interactive element in any form flow, and its states need to communicate availability, progress, and outcome clearly to every user, including those navigating by keyboard.

Following Adam Silver's Form Design Patterns: one primary action per form, the submit button label names the outcome ("Submit intake form", not just "Submit"), and the button is never pre-emptively disabled — inline validation guides users to a submittable state rather than leaving them with a greyed-out button and no explanation. The focus ring on the submit button is the same 2px black ring used across all interactive elements, ensuring a consistent and predictable keyboard experience throughout the form.

Platform Infrastructure

Native component explorer: closing the Storybook gap on iOS & Android

Storybook gave web teams a live sandbox for every component state, variant, and edge case. Our native platforms had no equivalent — iOS and Android components existed only in static specs, screenshots, and tribal knowledge passed at handoffs. That gap wasn't just inconvenient: designers couldn't experience real native behavior, developers lacked a shared reference during handoff conversations, and accessibility testing happened in production rather than at the component level.

I partnered with the iOS and Android systems engineers to build beta "component explorer" applications — native apps functioning as a live design system catalogue, letting designers, engineers, and QA testers interact with real components on real devices.

The design system served three platforms simultaneously. Only one had interactive documentation that people could actually use.

What this meant in practice: designers formed mental models of native components from PDFs and Figma files — not the components themselves. Handoff conversations relied on screen recordings that couldn't be queried. And accessibility audits happened post-engineering, not at the component level where fixes cost the least.

The solution required a genuine engineering partnership, not just a design request. I couldn't build native apps alone — iOS and Android engineers owned the implementation layer. I made the case, defined capability requirements from the design system's perspective, contributed the component metadata and documentation content, and worked alongside both platform teams through testing and validation before internal rollout.

What we built

Over several sprints, the iOS and Android systems engineers each built beta "component explorer" applications — distributed internally via TestFlight and Firebase App Distribution. These weren't prototypes. They were live applications built on the same component implementations that shipped in production. Adding a new component to the design system meant adding it to the explorer on the same timeline.

Three things changed after the explorer apps rolled out — not just in process, but in the quality of work the design system could support.

🎨
Deeper designer fluency with native behavior

Designers moved from speccing components based on how they look in Figma to understanding how they actually behave on-device — including platform-specific animations, press states, haptic timing, and dynamic type reflow. Component critique sessions became markedly more specific, and design decisions better reflected native affordances rather than web mental models projected onto a phone screen.

🤝
A shared reference that elevated handoff conversations

With a common live reference, handoff conversations shifted from "here's what I'm thinking" to "here's the component, here's the state in question." Designers and engineers could open the same explorer app, navigate to the same component, and work through behavior from shared ground — rather than across the gap between a Figma spec and someone's memory of the implementation.

♿
Earlier, more thorough accessibility testing

The VoiceOver and TalkBack audit environment built into the explorer apps meant accessibility testing could happen at the component level, before a single product screen had been built. Both design and engineering could now test with assistive technology against isolated components — shifting accessibility from a final QA pass to a component design standard that engineering inherited by default.

What this demonstrated about systems leadership

  • Identifying the gap required seeing the design system from all three platform perspectives simultaneously — not just the one with the most existing tooling.
  • The solution required credibility with native engineering teams, not just design authority. The partnership was the product: without the iOS and Android engineers, there was no explorer.
  • Distributing component documentation as a live, installable application changed how the system was perceived — from a library teams were handed, to a platform teams worked within.

Accessibility isn't a checklist I apply at the end — it's a practice I've built systems around, taught publicly, and contributed to at the standards level. Two pieces of public work that show this in action.

Conference Talk · axe-con 2022

Building Accessible Design Systems: Try, Try, Try Again

DH
Co-presented with
Daniel Henderson-Ede ↗ — Daniel's write-up of this talk

Accessibility in a design system doesn't happen once — it's a practice of iteration, advocacy, and learning from what doesn't work the first time. At axe-con 2022, Daniel Henderson-Ede and I co-presented on what embedding accessibility into a large-scale enterprise design system actually looks like from the inside.

The talk was honest about the friction. Building for a Fortune 500 organization means navigating a lot of competing stakeholders, and the accessibility team doesn't always win the argument immediately. We shared what we learned from those losses: how to build the case over time, how to recover from decisions that don't go your way, and how to design systems resilient enough to hold inclusive standards even when implementation teams work at speed.

The session title — Try, Try, Try Again — was deliberately chosen. It captures the reality of doing this work inside large organizations, and it's the posture I bring to every system I build: persistence, iteration, and a commitment to getting the outcome right rather than getting credit for the first attempt.

What the talk covered

We structured the session around three challenges that come up repeatedly when building accessible design systems at scale: getting buy-in before there's proof, handling rollbacks and regressions without losing momentum, and creating documentation that actually changes behavior — not just records it.

The audience left with a framework for making the accessibility case to product and engineering leaders, a practical approach to auditing and triaging components by accessibility impact, and concrete examples of what "accessibility as a system principle" looks like in practice versus on paper.

Published Article · 2024

The path to WCAG 2.2 conformance through a design system

Published on
CVS Health Tech Blog · Medium
Read the full article ↗

When WCAG 2.2 was published as a W3C recommendation in October 2023, most design system teams faced months of retroactive remediation. Ours didn't. This published piece — co-authored with colleagues from the inclusive design team — documents the strategy behind that: future-proofing components against standards that hadn't been ratified yet.

The argument is architectural, not just technical. By building to the WCAG 2.2 working draft from day one — accepting the risk of normative language changes — we absorbed the cost of conformance work early and continuously, rather than in one disruptive, high-stakes sprint when the standard landed.

The strategic decision that saved months of rework

Most teams build to what's required today. We built to what we knew was coming: the intent of each criterion doesn't change even when the normative language does. Participating in the W3C Accessibility Guidelines Working Group gave our team direct visibility into the trajectory of WCAG — and even without that access, the public working drafts contain enough signal to make informed bets.

Of the 9 new success criteria introduced in WCAG 2.2, 8 applied directly to our system. The article walks through each one — from focus appearance and target size minimums to accessible authentication and redundant entry — and documents the specific component decisions we made to conform. In every case, the work had already been done before the standard was published.

Why this matters for system governance

This wasn't just an accessibility win — it was a demonstration of what principled system governance looks like. The principle was simple: conformance is a floor, not a ceiling. We documented it, operationalized it, and built it into the component lifecycle. The WCAG 2.2 moment was when that investment paid off publicly.

AI & the future of
design systems

In early 2026, I attended the Into Design Systems AI Design Systems Conference — 21 speakers from WhatsApp, GitHub, Figma, Adobe Spectrum, Miro, and more. Here's what I brought back.

Into Design Systems 2026

Design systems as AI infrastructure — what I learned

The conference's central thesis was clear: the future of design systems isn't just about humans consuming them — it's about machines. Every session circled back to the same insight: a well-structured design system is the only thing that gives AI context it can't get from training data alone.

These are the four frameworks I came home with, and how they're shaping the way I think about systems work today.

01

Machine-readable documentation is the new API

Structuring component metadata and design decisions as machine-readable formats — not just human-readable docs — transforms a design system into infrastructure that AI tools can reason about reliably.

Diana Wolosin (Indeed) — MCP benchmark testing across 8 documentation configurations
02

Governance becomes an automated output, not a manual process

Encoding design contracts as rules and instructions means agents enforce consistency as they generate — reducing the 30–40% of maintenance time most teams spend just keeping things from drifting.

Romina Kavcic (The Design System Guide) — Agentic Design Systems
03

Your system's decisions are the guardrails AI can't hallucinate

AI can't get your team's specific architectural decisions from training data. But it can get them from a well-structured system. Context-based design systems turn your hard-earned best practices into AI guardrails.

Brad Frost, TJ Pitre & Ian Frost — Context-Based Design Systems to the Rescue
04

The handoff model is being reinvented — incrementally

Teams starting with MCP integration and incrementally expanding AI capabilities — rather than big-bang transformations — are seeing real velocity gains without quality loss. The path is additive, not disruptive.

Andressa Lombardo & Eddie Machado (Miro) — The Path to an AI-Enabled Design System

How this maps to where I want to work next

  • Machine-readable documentation structures directly enable Figma MCP server integration — the exact bridge between design and production code that fast-moving health tech teams need.
  • Agentic governance frameworks address the fragmentation risk that comes with high-velocity, distributed product teams — keeping a system coherent without creating bottlenecks.
  • The incremental MCP path means I can start delivering AI-enabled system velocity within weeks of joining, not after a multi-quarter transformation project.
  • Building AI infrastructure on top of a strong accessibility foundation ensures the speed gains don't come at the cost of inclusive design — non-negotiable in healthcare.

My current practice

I'm actively applying these frameworks in my work: structuring token documentation in markdown with semantic metadata, building component specifications that are AI-parseable, and exploring Figma MCP server configurations to reduce the manual overhead in design-to-code handoff.

The goal isn't to replace the design judgment in a system — it's to make that judgment durable and scalable, so the system serves the next engineer who joins the team just as well as the one who built it.
